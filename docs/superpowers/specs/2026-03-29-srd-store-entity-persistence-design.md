# SRD Store + Entity Persistence + UI Polish

## Goal

Bundle all D&D 5e SRD content into the Archivist Obsidian plugin, enable saving AI-generated entities as vault notes with cross-referencing via Obsidian wikilinks, and polish the chat UI (icon sizes, fix broken dropdowns, add entity autocomplete).

## Scope

This is **Spec 1** of a two-part design:
- **Spec 1 (this):** SRD data store, entity persistence, Copy & Save, UI polish, `[[` autocomplete in AI chat
- **Spec 2 (future):** `[[` autocomplete in Obsidian editor, hover preview popups, editor integration

## 1. SRD Data Store

### What It Does

Bundles all Open5e SRD data as JSON files in the plugin. On plugin load, parses them into in-memory Maps for O(1) lookup by slug and fast substring search for autocomplete.

### Data Sources

Copy JSON from the original Archivist app at `/server/data/srd/*/all.json`. Supported entity types:

| Type | Source Path | Approximate Count |
|------|------------|-------------------|
| Monsters | `monsters/all.json` | 400+ |
| Spells | `spells/all.json` | 300+ |
| Magic Items | `magicitems/all.json` | 200+ |
| Armor | `armor/all.json` | ~13 |
| Weapons | `weapons/all.json` | ~37 |
| Equipment | `equipment/all.json` | ~50 |
| Feats | `feats/all.json` | ~40 |
| Conditions | `conditions/all.json` | 14 |
| Classes | `classes/all.json` | ~12 |
| Backgrounds | `backgrounds/all.json` | ~15 |

### Architecture

**Files:**
- `src/srd/data/` -- bundled JSON files, one per entity type
- `src/srd/srd-store.ts` -- singleton SRD store class

**SrdStore class:**
```typescript
interface SrdEntity {
  slug: string;        // unique identifier (from Open5e)
  name: string;
  type: string;        // "monster" | "spell" | "magic-item" | "armor" | "weapon" | "equipment" | "feat" | "condition" | "class" | "background"
  data: Record<string, unknown>;  // full raw data
}

class SrdStore {
  private entities: Map<string, SrdEntity>;        // slug -> entity
  private byType: Map<string, SrdEntity[]>;        // type -> entities of that type

  load(): void;                                     // parse bundled JSON into Maps
  getBySlug(slug: string): SrdEntity | undefined;
  search(query: string, type?: string): SrdEntity[];  // case-insensitive substring on name, max 20 results
  getAllOfType(type: string): SrdEntity[];
  getTypes(): string[];
}
```

- On plugin load, `SrdStore.load()` parses all JSON files into the Maps.
- Memory footprint: ~2-5MB for full SRD (acceptable for an Obsidian plugin).
- Search returns results ranked by: exact name match first, then starts-with, then contains.

## 2. Entity Persistence (Vault Notes)

### What It Does

Saves entities (both SRD and AI-generated/custom) as Obsidian markdown notes with YAML frontmatter containing full structured data. On plugin load, scans entity folders and loads them into a unified registry.

### Folder Structure

```
Compendium/
  SRD/
    Monsters/
      Ancient Red Dragon.md
      Goblin.md
      ...
    Spells/
      Fireball.md
      Shield.md
      ...
    Magic Items/
      Bag of Holding.md
      ...
    Armor/
    Weapons/
    Equipment/
    Feats/
    Conditions/
    Classes/
    Backgrounds/
  me/                          <-- configurable, default "me"
    Monsters/
      Shadow Wyrm.md           <-- AI-generated
    Spells/
      Eldritch Blast Variant.md
    Items/
      Sword of the Phoenix.md
```

- `Compendium/SRD/` -- auto-generated from bundled SRD data on first load
- `Compendium/me/` -- user entities (AI-generated or custom), folder name configurable in settings
- Both root folder (`Compendium`) and user subfolder (`me`) are configurable in plugin settings

### Note Format

```yaml
---
archivist: true
entity_type: monster
slug: ancient-red-dragon
name: Ancient Red Dragon
source: srd
data:
  size: Gargantuan
  type: dragon
  alignment: chaotic evil
  ac:
    - value: 22
      type: natural armor
  hp:
    average: 546
    formula: 28d20 + 252
  # ... full stat block data
---
# Ancient Red Dragon
*Gargantuan dragon, chaotic evil*

AC 22 (natural armor) | HP 546 (28d20 + 252) | CR 24
```

- `archivist: true` -- identifies notes managed by the plugin
- `entity_type` -- the entity category
- `slug` -- unique identifier, kebab-case
- `source` -- `"srd"` or `"custom"`
- `data` -- full structured entity data (same format as SRD JSON)
- Markdown body below frontmatter is a minimal human-readable summary (name, key stats). Full stat block rendering from `data` via a markdown post-processor is deferred to Spec 2 (editor integration). In Spec 1, entity notes are readable but plain.

### Slug Uniqueness

- SRD slugs come directly from Open5e data (e.g., `ancient-red-dragon`)
- User entity slugs are auto-generated from name as kebab-case
- If a user creates an entity with the same name as an SRD entity, the slug gets a `-custom` suffix (e.g., `ancient-red-dragon-custom`)
- If that also exists, numeric suffixes: `-custom-2`, `-custom-3`, etc.
- Uniqueness is enforced by slug, not name. Two entities can share a name if they have different slugs.

### First-Load SRD Import

On first plugin load (or when `Compendium/SRD/` doesn't exist):

1. Plugin detects missing SRD folder
2. Shows Obsidian notice with progress: "Importing SRD content... 423/1081"
3. Creates all type subfolders under `Compendium/SRD/`
4. Generates one `.md` file per SRD entity with frontmatter + summary body
5. Completes with notice: "SRD import complete. 1081 entities added to vault."

If interrupted (e.g., plugin reload), resumes by checking which files already exist.

### Entity Registry

**File:** `src/entities/entity-registry.ts`

```typescript
interface RegisteredEntity {
  slug: string;
  name: string;
  type: string;          // "monster" | "spell" | "magic-item" | etc.
  source: "srd" | "custom";
  filePath: string;      // vault-relative path to the .md note
  data: Record<string, unknown>;
}

class EntityRegistry {
  private entities: Map<string, RegisteredEntity>;  // slug -> entity

  load(): Promise<void>;          // scan Compendium/ folders, parse frontmatter
  getBySlug(slug: string): RegisteredEntity | undefined;
  search(query: string, type?: string): RegisteredEntity[];  // max 20, ranked
  save(entity: RegisteredEntity): Promise<string>;           // create vault note, return file path
  getTypes(): string[];
}
```

- Merges SRD and user entities into one Map
- Search returns results with source indicator so autocomplete can show "SRD" vs "Custom" badge
- No override behavior -- both coexist, differentiated by slug

### Source Indicator on Entity Blocks

Rendered entity blocks (in chat and in note previews) show a small source badge:
- SRD entities: "SRD" badge (subtle, uses `book-open` Lucide icon)
- Custom entities: "Custom" badge (uses `pen-tool` Lucide icon)
- Badge appears in the entity block header, next to the entity name

## 3. Copy & Save

### What It Does

Single button on AI-generated entity blocks in chat. Copies the entity as YAML to clipboard AND saves it as a vault note in the user's entity folder.

### Behavior

1. User clicks "Copy & Save" button on an entity block in chat
2. Plugin copies entity data as YAML to clipboard (existing behavior)
3. Plugin generates slug from entity name
4. Plugin checks for slug uniqueness, appends suffix if needed
5. Plugin creates vault note at `Compendium/me/{Type}/{Name}.md`
6. Plugin adds entity to the EntityRegistry
7. Shows Obsidian notice: "Saved to Compendium/me/Monsters/Shadow Wyrm.md"

### Button Design

- Replaces the current "Copy to Clipboard" button on entity blocks
- Label: "Copy & Save" with `copy-plus` Lucide icon
- Same position and styling as current copy button
- After click, briefly shows "Saved!" confirmation state

### Add `[[` Entity Autocomplete in AI Chat

New dropdown in the AI chat textarea triggered by `[[`.

**Trigger:** User types `[[` in the textarea.

**Type prefix filtering:**
- `[[` -- shows all entities (SRD + custom) + vault files
- `[[monster:` -- filters to monsters only
- `[[spell:` -- filters to spells only
- `[[item:` -- filters to items (magic items + equipment + armor + weapons)
- `[[doc:` -- filters to vault documents only
- Other type prefixes: `feat:`, `condition:`, `class:`, `background:`

**Dropdown behavior:**
- Positioned above textarea (same as `@` and `/` dropdowns)
- Max 20 results
- Each result shows: type icon + name + source badge ("SRD" or "Custom" or "Doc")
- Keyboard navigation: Arrow keys, Enter/Tab to select, Esc to close
- Debounced filtering (200ms)

**On selection:**
- Inserts `[[monster:Ancient Red Dragon]]` (with type prefix) into textarea
- Attaches the full entity stat block as context for the AI message
- The AI receives the entity data alongside the user's message text so it can reason about the entity

**Context injection:**
- When message is sent, any `[[type:name]]` references in the text are resolved via EntityRegistry
- Full entity data is appended to the user message as a structured context block (not visible in chat UI, but included in the API call)
- Format: each referenced entity's data is serialized as YAML and appended after the user's text, wrapped in a `<entity-context>` tag with the entity name and type

**File:** `src/ui/components/entity-autocomplete.ts` -- new class, same pattern as MentionDropdown
